## Introduction: The Last Mile Problem

I've been building AI agents with the Strands SDK and deploying them to AWS Bedrock AgentCore. The developer experience is excellent — you define your agent's behavior, deploy it to AgentCore, and you have a managed runtime that handles scaling, versioning, and invocation. The agent *works*.

But then someone asks: "Can I try it?"

And you realize you have nowhere to send them. AgentCore gives you an API. It doesn't give you a login page, a chat interface, or any way for a non-developer to discover and interact with your agents. You need a frontend. And building a frontend means authentication, hosting, access control, streaming response handling, session management — suddenly your "quick demo" is a multi-week project.

I built Aura because I got tired of that gap. I wanted to go from "agent deployed on AgentCore" to "anyone on my team can use it" in under ten minutes. Not ten days. Ten minutes.

## What Aura Does

Aura is an open-source framework that gives you a complete, production-ready web frontend for AI agents running on AWS Bedrock AgentCore. It ships as three artifacts: a CloudFormation template, a prebuilt React frontend bundle, and a README. You deploy the template, and you get:

- **Passwordless sign-in** — users authenticate with email OTP. No passwords to manage, no password reset flows to build.
- **Agent dashboard** — authenticated users see the agents available to them and start conversations instantly.
- **Streaming chat** — real-time token-by-token responses from AgentCore agents and harnesses, rendered as they arrive.
- **Admin console** — register agents by ARN, control who can access what, manage user groups.
- **Consent capture** — terms and privacy acceptance before first use, because production apps need governance.

The key insight: the same prebuilt frontend bundle works for every deployment. Configuration is injected at runtime via a `config.json` generated by the stack. No rebuilding, no environment variables baked into the bundle, no CI/CD pipeline required.

## Architecture

[DIAGRAM: architecture-overview]

Aura's architecture is deliberately minimal — every resource exists because removing it would break a user-facing capability.

```
User → Amplify Hosting (React SPA)
         → Cognito User Pool (Email OTP Authentication)
         → Cognito Identity Pool (IAM Credential Vending)
         → DynamoDB (Agent Registry — availability + access rules)
         → AgentCore (Streaming Agent Invocation)
```

**Amazon Cognito User Pool** handles authentication. It's configured for the ESSENTIALS tier with `EMAIL_OTP` as the sole authentication factor. No passwords exist in the system — users receive a one-time code via email and that's it.

**Cognito Identity Pool** maps authenticated users to IAM roles. Users in the `admin` group assume an admin role; everyone else gets the standard user role. This is where the magic happens: IAM credentials issued by the Identity Pool are what authorize direct calls to AgentCore and DynamoDB from the browser.

**DynamoDB** stores the agent registry — a table of AgentCore runtime ARNs, display names, and access rules. Each entry specifies whether an agent is available to all authenticated users or restricted to specific emails/groups.

**AWS Amplify Hosting** serves the React SPA. The frontend bundle is deployed once during stack creation via a Lambda-backed custom resource that pulls `frontend.zip` from the GitHub release.

**IAM Roles** are scoped tightly. The user role permits `bedrock-agentcore:InvokeAgentRuntime` and `dynamodb:Query` on the registry table. The admin role adds `cognito-idp:Admin*` operations and `dynamodb:PutItem`/`DeleteItem` on the registry.

**Lambda Functions** serve as CloudFormation custom resources: one seeds the initial admin account into Cognito, another deploys the frontend bundle to Amplify. They run only during stack create/update — no Lambda executes at runtime.

## How It Works

[DIAGRAM: user-journey]

### Deploy (Admin — Once)

The administrator deploys the CloudFormation template with three parameters: `ProjectName`, `AdminEmail`, and optionally `CustomDomain`. Five to ten minutes later, every resource exists. The admin receives their first OTP to sign in.

### Sign In (User — Every Session)

A user navigates to the Amplify URL, enters their email, receives an OTP, and enters it. Behind the scenes, Cognito authenticates them, the Identity Pool issues temporary IAM credentials scoped to their group membership, and the SPA caches these credentials for the session duration (with automatic refresh).

### Discover Agents (User — Dashboard)

The SPA queries DynamoDB directly (using the IAM credentials from the Identity Pool) to fetch agents the user is authorized to see. The query filters on the user's email and Cognito group memberships against each agent's access rules. No backend API needed — the access control logic lives in the data model and IAM policy.

### Chat (User — Streaming)

When a user selects an agent and sends a message, the SPA calls `InvokeAgentRuntime` directly via the AWS SDK for JavaScript. The request includes:

- `runtimeSessionId` — stable per login session for conversation continuity
- `runtimeUserId` — the user's Cognito `sub` (opaque UUID, never their email)
- `payload` — JSON containing the user's prompt

AgentCore streams the response back. The SPA renders tokens as they arrive, giving users the real-time experience they expect from modern AI interfaces.

### Manage (Admin — Ongoing)

Admins register new agents by providing their AgentCore runtime ARN and a display name. They set availability: open to all users, or restricted by email address and/or access group. They create groups, add members, and the access model updates immediately — no redeployment, no config changes.

## The Technical Decisions

### Why Passwordless?

Passwords are liability. They need to be hashed, stored, rotated, and inevitably reset. For an internal tool or a team-facing agent platform, email OTP eliminates an entire category of security concerns. Cognito's ESSENTIALS tier supports this natively — no custom Lambda triggers, no pre/post-authentication hooks. The trade-off (users need email access) is acceptable for the use case.

### Why Direct SDK Calls from the Browser?

Traditional architectures put an API Gateway + Lambda between the frontend and backend services. I skipped that layer intentionally. Cognito Identity Pool already vends IAM credentials scoped to exactly what the user should access. DynamoDB queries and AgentCore invocations happen directly from the browser with those credentials. This eliminates:

- Cold start latency on the chat path
- An API layer to maintain, version, and secure
- Double-hop network latency
- Additional cost per invocation

The IAM policies are the API contract. If your credentials allow `InvokeAgentRuntime` on a specific agent ARN, you can call it. If they don't, you can't. The authorization model is the infrastructure.

### Why a Single CloudFormation Template?

I wanted deployment to be one action. Not "deploy the backend stack, then configure the frontend, then set up the auth." One template. One `aws cloudformation deploy` command. Done.

CloudFormation custom resources (backed by Lambda) handle the bootstrapping that can't be expressed declaratively: seeding the admin account and deploying the frontend bundle. The result is a self-contained stack that provisions everything from nothing and removes everything on deletion.

### Why Amplify Hosting Over S3 + CloudFront?

Amplify Hosting gives me HTTPS, managed certificates, branch-based deployments, and custom domain support with less template complexity than an S3 bucket + CloudFront distribution + ACM certificate + Route53 records. For a single-page app with no server-side rendering, it's the right trade-off.

## Getting Started

### Prerequisites

- An AWS account with CloudFormation, Cognito, DynamoDB, Amplify, IAM, and Lambda permissions
- At least one AgentCore agent deployed and configured to accept IAM invocation

### Deploy

**Console:**

1. Go to CloudFormation → Create stack → Upload `template.yaml`
2. Set `ProjectName` (e.g., `myteam`), `AdminEmail`, and optionally `CustomDomain`
3. Acknowledge IAM resource creation → Create stack
4. Wait ~5-10 minutes

**CLI:**

```bash
aws cloudformation deploy \
  --template-file template.yaml \
  --stack-name aura-myteam \
  --parameter-overrides \
    ProjectName=myteam \
    AdminEmail=admin@yourcompany.com \
  --capabilities CAPABILITY_NAMED_IAM
```

### Configure Your Agent

Your AgentCore agent needs to allow invocation from the IAM role created by the stack:

```
arn:aws:iam::{account-id}:role/{project-name}-user-role
```

Grant it `bedrock-agentcore:InvokeAgentRuntime` permission on your agent's ARN.

### Register and Go

Sign in as admin, navigate to the Admin Console, paste your agent's ARN, give it a name, set availability, and click Register. Your users can now discover and chat with your agent.

## What's Next

Aura is a starting point, not a finished product. The roadmap includes:

- **Federated SSO** — SAML/OIDC integration for enterprise identity providers
- **Conversation history** — persist and resume past conversations across sessions
- **Agent analytics** — usage metrics, response quality tracking, cost attribution
- **Multi-region support** — deploy agents in multiple regions with a single control plane
- **Custom theming** — brand the interface without rebuilding the frontend
- **Agent chaining** — let users compose multi-agent workflows from the dashboard
