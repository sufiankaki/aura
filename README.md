# Aura — AgentCore Frontend Framework

A ready-to-deploy web frontend for AI agents running on **AWS Bedrock AgentCore**. Deploy a single CloudFormation template and get a complete platform: passwordless authentication, an agent dashboard, streaming chat, and admin controls.

Published as the CloudFormation template (`template.yaml`), a prebuilt frontend bundle (`frontend.zip` in GitHub releases), and this README.

## What It Does

- **Passwordless sign-in** — users authenticate with email OTP (no passwords)
- **Agent dashboard** — see available agents and start conversations
- **Streaming chat** — real-time responses from AgentCore agents
- **Admin console** — register agents, control access per user or group
- **Consent capture** — terms & privacy acceptance before first use
- **One-click deploy** — single CloudFormation stack provisions everything

## Use Case

You've built AI agents with the Strands SDK and deployed them to AgentCore. Now you need a way for end users to access them. Aura gives you a self-service web app — deployed in minutes — so your users can discover and interact with your agents without any custom frontend development.

## Prerequisites

- An **AWS account** with permissions to deploy CloudFormation stacks (Cognito, DynamoDB, Amplify, IAM, Lambda)
- At least one **AgentCore agent** deployed and ready to receive invocations
- Your AgentCore agent must be configured to **allow IAM invocation** from the Cognito Identity Pool credentials (see [Agent Configuration](#configuring-your-agent-for-aura) below)

## Stack Parameters

| Parameter | Required | Description |
|-----------|----------|-------------|
| `ProjectName` | Yes | Prefix for all resource names. 3-30 characters: lowercase letters, digits, hyphens. Must start with a letter, cannot end with a hyphen. |
| `AdminEmail` | Yes | Email address for the initial administrator account. This person gets admin + user group membership on first deployment. |
| `CustomDomain` | No | Custom domain FQDN (e.g., `agents.yourcompany.com`). Leave blank to use the default Amplify domain. |

## Deployment

### Step 1: Deploy the Stack

**AWS Console:**
1. Go to **CloudFormation** → **Create stack** → **With new resources**
2. Upload `template.yaml` or provide the template URL
3. Fill in the stack parameters (ProjectName, AdminEmail, optionally CustomDomain)
4. Acknowledge IAM resource creation
5. Click **Create stack**

**AWS CLI:**
```bash
aws cloudformation deploy \
  --template-file template.yaml \
  --stack-name aura-myproject \
  --parameter-overrides \
    ProjectName=myproject \
    AdminEmail=admin@example.com \
  --capabilities CAPABILITY_NAMED_IAM
```

### Step 2: Wait for Completion

The stack creates all resources (~5-10 minutes):
- Cognito User Pool with email OTP sign-in
- Identity Pool with role mapping
- DynamoDB agent registry table
- Amplify Hosting application
- Seeds the initial admin account
- Deploys the prebuilt frontend

### Step 3: Find Your Stack Outputs

After completion, check the **Outputs** tab:

| Output | Description |
|--------|-------------|
| `ApplicationUrl` | Public HTTPS URL of your deployed frontend |
| `UserPoolId` | Cognito User Pool identifier |
| `AppClientId` | Cognito App Client identifier |
| `IdentityPoolId` | Identity Pool identifier |
| `AgentRegistryTableName` | DynamoDB table name |
| `AmplifyAppId` | Amplify application ID |

### Step 4: (Optional) Configure Custom Domain

If you provided a `CustomDomain`, go to the Amplify Console for your app and complete DNS verification by adding the CNAME records shown to your DNS provider.

## How It Works

The prebuilt frontend bundle is deployed to Amplify Hosting during stack creation. At runtime, the frontend loads a `config.json` file (generated by the stack) containing the Cognito, DynamoDB, and region identifiers. This means the same frontend bundle works for any deployment — no rebuilding required.

**Runtime Config** contains:
- Cognito User Pool ID
- App Client ID
- Identity Pool ID
- Agent Registry table name
- AWS Region

## Configuring Your Agent for Aura

Your AgentCore agent must accept invocations from IAM credentials issued by the Cognito Identity Pool.

**Required settings on your agent:**
1. The agent must allow the IAM principal `arn:aws:iam::{account-id}:role/{project-name}-user-role` to call `bedrock-agentcore:InvokeAgentRuntime`
2. The agent must accept `runtimeSessionId` and `runtimeUserId` headers for session/user context

**What Aura passes to your agent:**
- `runtimeSessionId` — unique per login session (stable across token refreshes)
- `runtimeUserId` — Cognito `sub` (opaque UUID, never the user's email)
- `payload` — JSON with `{ "prompt": "user's message" }`

## Admin Tasks

### Registering Agents

1. Sign in as the admin (use the email from `AdminEmail` parameter)
2. Navigate to **Admin Console** → **Agents** tab
3. Enter the AgentCore agent ARN and a display name
4. Set availability:
   - **All users** — everyone who signs in can see and use this agent
   - **Restricted** — specify email addresses and/or access group names
5. Click **Register Agent**

### Managing Access Groups

1. Go to **Admin Console** → **Access Groups** tab
2. **Create a group** — enter a name (this creates a Cognito group)
3. **Add users** — enter email addresses to add to the group
   - If the user doesn't have an account, one is created automatically
4. **Remove users** — click Remove next to their email
5. Use group names in agent availability rules to grant access to sets of users

## Operations

After deployment, the platform administrator can:

- **Register/remove agents** via the Admin Console
- **Manage access groups** (create groups, add/remove members)
- **Monitor sign-ups** via the Cognito Console
- **View usage** via CloudWatch (Lambda logs, DynamoDB metrics)
- **Update the frontend** by creating a new GitHub release and triggering a stack update
- **Delete the stack** to remove all resources (CloudFormation rollback removes everything)
- **Scale** — DynamoDB uses on-demand billing, Cognito and Amplify scale automatically

## Architecture

```
User → Amplify (SPA) → Cognito (Email OTP)
                      → Identity Pool (IAM Creds)
                      → DynamoDB (Agent Registry)
                      → AgentCore (Streaming Chat)
```

All resources are created in the deployment region. No cross-region calls.

## Updating

To update the frontend:
1. Build the frontend (`cd frontend && npm run build`)
2. Zip the `dist/` folder as `frontend.zip`
3. Create a new GitHub release with the zip attached
4. Run a CloudFormation stack update (the custom resource re-deploys from the latest release)

## Troubleshooting

| Issue | Solution |
|-------|----------|
| Stack creation fails | Check the Events tab for the failed resource. Common: insufficient IAM permissions. |
| Can't sign in | Verify the User Pool is in ESSENTIALS tier and EMAIL_OTP is enabled. |
| Agent not showing | Check the agent's availability rule. Ensure the user's email or group matches. |
| Chat returns error | Verify the agent allows IAM invocation from the Identity Pool role. |
| Custom domain not working | Complete DNS verification in the Amplify Console. |
